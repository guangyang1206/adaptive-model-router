# MVP-3 Implementation Design — `@adaptive-router/control-plane`

> Status: **Phase 2 implementation design** (architect: 高见远). Derived strictly
> from `docs/spec-mvp3.md` §3/§4/§5 and the **real** signatures read from
> `packages/sdk/src/*` and `packages/dashboard/src/index.ts`. This document
> guides backend development; it contains structure + SQL/config drafts, not

## 0. Ground-truth corrections (read the code, don't trust the doc)

Three facts from the actual source that the Spec/architecture text glossed over.
Developers must build against **these**, not the prose:

1. **`DashboardDataSource` / `createReadOnlyDataAccess` live in
   `@adaptive-router/dashboard`, NOT the SDK.** (Spec §3 reuse table listed them
   under "SDK export" — that is imprecise.) The SDK exports the *types*
   (`RouterTrace`, `StoredRequest`, `ModelProfile`, `Mvp2StoreExtension`,
   `ExtendedTraceStore`) and the readers (`getEvalsOverview` etc.). The
   `DashboardDataSource` **shape** (`listTraces()/listModels()/store?`) and the
   `createReadOnlyDataAccess()` factory that turns it into the 12-endpoint
   `DashboardDataAccess` are dashboard exports.
   → **control-plane depends on BOTH `@adaptive-router/sdk` and
   `@adaptive-router/dashboard`** (both `workspace:*`, both zero-runtime-dep, so
   the boundary is not violated).

2. **The response envelope is `{ code, data, message }` where `code` is the
   STRING `"OK"` / `"ERROR"`, not the numeric `0`.** Proof: `sendJson()` in
   `packages/dashboard/src/index.ts` emits
   `{ code: statusCode>=400 ? "ERROR" : "OK", data, message }`, and the CI
   dashboard smoke test asserts `summary.code !== 'OK'` to fail. Spec §4 says
   `code:0`. **We must not break the dashboard smoke test.** Resolution below
   (§4.1). New control-plane endpoints reuse the SAME string-code envelope.

3. **The `DashboardDataSource.listTraces()` feed type is
   `DashboardTrace`, and `listModels()` is `DashboardModel`** (dashboard-local
   types), which are structurally the fields the reader needs — NOT the SDK
   `RouterTrace`/`ModelProfile` verbatim. They are close but not identical (e.g.
   `DashboardTrace` has `estimatedCostUsd` + `usage`, `DashboardModel` nests
   `health.status`/`health.latencyP50Ms` and `cost.inputPer1M`). Our PG mapping
   targets the **dashboard** types (§5).

None of these change the architecture; they change which package we import from
and the exact mapper output types. They are the difference between "compiles
first try" and "three hours of type errors."

### 1.4 `node-shims.d.ts` convention

Hand-declare ONLY the `node:*` surface actually used, mirroring
`packages/dashboard/src/node-shims.d.ts`. control-plane needs more than the
dashboard: `node:http` (fuller — request body streaming, headers), `node:crypto`
(token hashing), `process.env`, `node:process`. Draft:

```ts

  export type IncomingMessage = {
    url?: string
    method?: string
    headers: Record<string, string | string[] | undefined>
    on(event: "data" | "end" | "error", cb: (chunk?: unknown) => void): void
  }
  export type ServerResponse = {
    statusCode: number
    setHeader(name: string, value: string): void
    getHeader(name: string): string | number | string[] | undefined
    end(data?: string): void
    writeHead(status: number, headers?: Record<string, string>): void
  }
  export type Server = { close(cb?: (e?: Error) => void): void }
  export function createServer(
    listener: (req: IncomingMessage, res: ServerResponse) => void | Promise<void>,
  ): Server & { listen(port: number, host: string, cb?: () => void): void }
}

declare module "node:crypto" {
  export function randomBytes(size: number): { toString(enc: "hex" | "base64url"): string }
  export function createHash(algo: "sha256"): {
    update(data: string): { digest(enc: "hex"): string }

  export function timingSafeEqual(a: Uint8Array, b: Uint8Array): boolean

declare const process: { env: Record<string, string | undefined>; exit(code?: number): never }
```

Better-Auth and postgres.js ship their own `.d.ts`, so those don't need shims —
only the Node built-ins do. `skipLibCheck: true` (base config) keeps their types

## 2. Postgres migrations (Spec §5, hand-written SQL + version table)

### 2.1 `schema_migrations` version mechanism

```sql
-- Applied automatically before any app query on first boot (see 2.3).
CREATE TABLE IF NOT EXISTS schema_migrations (
  version     text PRIMARY KEY,     -- e.g. '0001_init'
  applied_at  timestamptz NOT NULL DEFAULT now()
);
```

Runner algorithm (`db/migrate.ts`):
1. `CREATE TABLE IF NOT EXISTS schema_migrations` (bootstrap).
2. Read the ordered list of `db/migrations/*.sql` (filename = version, sorted
   ascending — the numeric prefix guarantees order).
3. `SELECT version FROM schema_migrations` → set of applied.
4. For each not-yet-applied file, **inside a transaction**: run the file's SQL,
   then `INSERT INTO schema_migrations(version) VALUES ($1)`. Commit. Any error
   → rollback that file and abort (fail loud; do not silently skip).
5. Migrations are append-only and never edited after merge (same ethos as the
   SDK's append-only JSONL).

> postgres.js note: `sql.file()` / `sql.unsafe(rawSql)` runs a full SQL file.
> Wrap each in `sql.begin(async sql => { ... })` for the per-file transaction.

### 2.2 `0001_init.sql` (draft — the 6 app tables from Spec §5)

Indexes are EXACTLY the Spec §5 set — no speculative additions.

```sql
-- 0001_init.sql — organizations, users are (partly) Better-Auth-owned; see 2.4.
-- We create the app-domain tables. `users` is created by Better-Auth's own
-- schema (it owns `user`, `session`, `account`, `verification`). To avoid a
-- name/ownership clash we reference Better-Auth's user table (see 2.4 decision).

CREATE TABLE IF NOT EXISTS organizations (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name        text NOT NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE IF NOT EXISTS memberships (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     text NOT NULL,          -- FK to Better-Auth user.id (text ids) — see 2.4
  org_id      uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  role        text NOT NULL DEFAULT 'member',   -- 'owner' | 'member' enforced; column allows more
  created_at  timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, org_id)
);
CREATE INDEX IF NOT EXISTS idx_memberships_user ON memberships(user_id);

CREATE TABLE IF NOT EXISTS projects (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name        text NOT NULL,
  slug        text NOT NULL,
  accent      text,                    -- one of the fixed 8-color palette
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now(),
  UNIQUE (org_id, slug)
);

CREATE TABLE IF NOT EXISTS ingest_tokens (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id    uuid NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  token_hash    text NOT NULL,         -- sha256 hex of the plaintext token; plaintext never stored
  created_at    timestamptz NOT NULL DEFAULT now(),
  last_used_at  timestamptz,
  revoked_at    timestamptz            -- soft delete / revocation
);
CREATE INDEX IF NOT EXISTS idx_ingest_tokens_project ON ingest_tokens(project_id);
-- token_hash lookups happen on the hot ingest path → unique index doubles as the lookup index:
CREATE UNIQUE INDEX IF NOT EXISTS idx_ingest_tokens_hash ON ingest_tokens(token_hash);

CREATE TABLE IF NOT EXISTS router_traces (
  trace_id            text PRIMARY KEY,
  project_id          uuid NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  decision_id         text NOT NULL,
  status              text NOT NULL,
  chosen_model        text,
  reason              text NOT NULL,
  latency_ms          integer,
  estimated           boolean NOT NULL DEFAULT true,
  estimated_cost_usd  real,
  input_tokens        integer,
  output_tokens       integer,
  total_tokens        integer,
  trace_json          jsonb NOT NULL,   -- full RouterTrace, forward-compat (append-only ethos)
  created_at          timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_traces_project_created ON router_traces(project_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_traces_project_status  ON router_traces(project_id, status);
```

Notes:
- `gen_random_uuid()` is built into Postgres 13+ (pgcrypto folded into core); no
  extension needed on PG 17. If targeting <13, `CREATE EXTENSION pgcrypto` — but
  Spec deploys `postgres:17`, so it's fine.
- `idx_ingest_tokens_hash` is a **UNIQUE** index: it enforces no duplicate hashes
  AND serves the O(1) ingest lookup. This is a Spec-implied FK/lookup index, not
  speculative.
- The two `router_traces` indexes are verbatim Spec §5. Nothing more.

### 2.3 First-boot auto-migration

`server.ts` bootstrap sequence (fail-fast, honest errors for A14):
```text
1. loadEnv()  → throws naming the missing var if DATABASE_URL / BETTER_AUTH_SECRET absent
2. connect postgres.js client (DATABASE_URL)
3. await runMigrations(sql)   ← schema_migrations runner (2.1); idempotent
4. build Better-Auth instance + mount handler
5. createServer(...).listen(PORT)
6. log "control-plane ready on :PORT"
```
Migrations run every boot but only apply unseen files → safe on restart (A8) and
on Render cold start. `/health` (P1) reports "migrations applied" by checking the
latest expected version is present in `schema_migrations`.

### 2.4 Better-Auth tables — decision

Better-Auth owns `user`, `session`, `account`, `verification` and (with the
`organization` plugin) can own `organization` + `member` + `invitation` tables.
**Decision to avoid double-owning the org model:**

- **Recommended (clean):** let Better-Auth's `organization` plugin own
  `organization`/`member`/`invitation`. Then our `projects` + `ingest_tokens` +
  `router_traces` reference `organization.id` and `user.id`. We DROP our own
  `organizations`/`memberships` tables and read org membership through
  Better-Auth's API/tables. This maps 1:1 to Spec's Org→Project→member model and
  removes a whole sync problem.
- **Alternative (Spec-literal):** keep our own `organizations`/`memberships`
  (as drafted in 0001) and use Better-Auth ONLY for `user`/`session`/`account`.
  Simpler mental model, but we hand-roll org CRUD + invites.

→ **My recommendation: use the Better-Auth `organization` plugin for
org/membership/invites** (less code, native invite flow for P1), and keep only
`projects`, `ingest_tokens`, `router_traces` as our hand-written tables. The
0001 draft above shows the Spec-literal option; if we adopt the plugin,
`organizations`/`memberships` are replaced by Better-Auth's generated tables and
our FKs point at `organization.id`/`user.id` (text ids).

**This is the one decision that needs team-lead/PRD sign-off** because it
slightly reinterprets Spec §5's "memberships" table as Better-Auth's `member`
table. Functionally identical (owner/member role, unique user×org); it just
moves ownership of the SQL. Generate Better-Auth's migration via its CLI
(`better-auth migrate` or `generate`) and commit the emitted SQL as
`0002_better_auth.sql` so it stays hand-reviewed and in the same versioned
runner — **not** an opaque runtime auto-migrate.

Migration `user.id` type: Better-Auth uses `text` ids by default → our
`memberships.user_id` / any user FK is `text` (reflected in 0001 draft).

## 3. Auth middleware + project-scope injection

### 3.1 Resolution flow: session → user → org → accessible projects

```text
Incoming request
  │
  ├─ path starts with /api/auth/  → hand entirely to Better-Auth handler (§4.2). Done.
  ├─ path === /ingest/traces      → ingest token auth (§4), NOT session. Done.
  ├─ path === /health             → public. Done.
  │
  └─ everything else (pages + reused /api/*):
       1. session = await auth.api.getSession({ headers })     // Better-Auth
       2. if !session:
            - HTML page request (Accept: text/html)  → 302 redirect to /login
            - API request                            → 401 + err("unauthenticated")
       3. user = session.user
       4. resolve accessible set:
            orgs      = memberships where user_id = user.id            (or Better-Auth member table)
            projects  = projects where org_id IN (orgs)                 → Set<projectId>
            role(org) = memberships.role                                 (owner|member)
       5. determine currentProjectId:
            - from path param (/api/orgs/:orgId/... , /api/projects/:projectId/...)
            - or from a `?project=` / cookie for the reused dashboard /api/* (the
              App Shell project switcher sets it)
       6. AUTHORIZE: if currentProjectId ∉ accessibleProjects  → 403 err("forbidden") (A6, no leak)
       7. attach { user, currentProjectId, accessibleProjects, role } to a request-scoped ctx
       8. dispatch to the route handler
```

The scope resolver (`auth/scope.ts`) is a pure function
`resolveScope(sql, userId) → { orgs, projects, roleByOrg }`, unit-testable
without HTTP (test/scope.test.mjs).

### 3.2 How `PgDataAccess` enforces `WHERE project_id = $current`

**The scope is injected at construction time, not per-query-guessed.** For each
request we build a `DashboardDataSource` bound to exactly one `project_id`:

```text
handler(ctx):
  const source = createPgDashboardDataSource(sql, ctx.currentProjectId)   // scope baked in
  const dataAccess = createReadOnlyDataAccess(source)                     // dashboard factory
  → route the reused /api/* through dataAccess (§5)
```

Because `createPgDashboardDataSource` closes over `currentProjectId`, EVERY
query it issues (`listTraces`, `listModels`, and the `store` methods) hard-codes
`WHERE project_id = $1`. There is no code path that can query without the
predicate — the project id is a constructor argument, not a query option a caller
could omit. This is the structural guarantee behind A4/A5/A6.

### 3.3 The specific 403 point

- **Point A (path-scoped mgmt API):** in the auth middleware step 6, before the
  handler runs — if the `:projectId`/`:orgId` in the path isn't in the user's
  accessible set → 403. (Covers `/api/projects/:id/tokens`,
  `/api/orgs/:id/*`.)
- **Point B (reused dashboard API):** the `currentProjectId` used to build the
  `DashboardDataSource` is validated against `accessibleProjects` in the SAME
  step 6. A request for another project's data can never construct a data source
  for it → the query is never issued → 403, never a 200 with leaked rows.
- **Org-level "All projects" view:** builds a data source over
  `project_id IN (accessibleProjects)` (a multi-project variant of
  `createPgDashboardDataSource`), so the aggregate still cannot exceed the
  membership set.

`owner`-only actions (create project, issue/revoke token, close registration,
invite) additionally check `ctx.roleByOrg[orgId] === 'owner'` → else 403. Only
`owner`/`member` are judged (Spec ruling ②; other role values ignored).

## 4. `POST /ingest/traces` implementation design

### 4.1 Envelope reconciliation (must resolve before coding)

Spec §4 mandates `{code:0}`; the shipped dashboard emits `{code:"OK"}` and the CI
smoke test enforces the string form. **Decision: keep the existing string-code
envelope (`code:"OK"|"ERROR"`) everywhere**, including new control-plane
endpoints, because (a) changing it breaks the dashboard smoke test and the reused
`/api/*` responses, and (b) the reused endpoints are wrapped, not rewritten. I
recommend the Spec text be read as "a `{code,data,message}` envelope where
`code` discriminates success/failure" — the shipped string form satisfies that.
`envelope.ts`:

```ts
export const ok  = (data: unknown, message = "") => ({ code: "OK",    data, message })
export const err = (message: string, data: unknown = null) => ({ code: "ERROR", data, message })
```

HTTP status still carries the real signal (200 / 401 / 403 / 400). Team-lead:
please confirm this reconciliation so QA (A10–A12) tests against `code:"OK"`, not
`code:0`.

### 4.2 Full ingest flow

```text
POST /ingest/traces
  Header: Authorization: Bearer <INGEST_TOKEN>
  Body:   RouterTrace JSON (the SDK's trace; client-supplied project id is IGNORED)

Handler:
  1. Parse `Authorization: Bearer <t>`. Missing/malformed → 401 err("missing bearer token").
  2. hash = sha256_hex(t)                              // node:crypto, §7 helper
  3. row = SELECT id, project_id, revoked_at FROM ingest_tokens WHERE token_hash = hash
     (uses UNIQUE idx_ingest_tokens_hash — O(1))
  4. if !row               → 401 err("unknown ingest token")   (A12)
  5. if row.revoked_at     → 403 err("ingest token revoked")   (A12)
  6. projectId = row.project_id                        // SERVER-DERIVED, never from body
  7. validate body is a RouterTrace-shaped object (traceId, decisionId, status,
     reason present; else 400 err("invalid trace payload")). Do NOT deep-trust;
     store the whole thing in trace_json regardless (append-only).
  8. INSERT INTO router_traces (trace_id, project_id, decision_id, status,
       chosen_model, reason, latency_ms, estimated, estimated_cost_usd,
       input_tokens, output_tokens, total_tokens, trace_json, created_at)
     VALUES (..., $projectId, ..., $body::jsonb, now())
     ON CONFLICT (trace_id) DO NOTHING     // idempotent re-delivery
  9. UPDATE ingest_tokens SET last_used_at = now() WHERE id = row.id   (fire-and-forget)
 10. → 200 ok({ traceId })                                  (A10)
```

Column extraction from the trace body (mirrors the SQLite store's mapping in
`storage.ts` so the shapes stay parallel):
`input/output/total_tokens` ← `trace.usage?.{inputTokens,outputTokens,totalTokens}`;
`estimated_cost_usd` ← `trace.estimatedCostUsd ?? trace.usage?.costUsd`;
`estimated` ← `trace.estimated`. The full object always lands in `trace_json`.

### 4.3 Token lifecycle (Spec §4.3 + §5)

- **Create** (`POST /api/projects/:id/tokens`, owner-only): generate
  `randomBytes(32).toString("base64url")` → plaintext (shown ONCE in the
  response, per onboarding/api-keys page); store only `sha256_hex(plaintext)` in
  `token_hash`. Response `ok({ token: plaintext, id, createdAt })` — the ONLY
  time plaintext leaves the server.
- **List** (`GET /api/projects/:id/tokens`, member): return
  `id, masked (e.g. "ark_live_••••" + last 4 of a stored hint OR just id),
  created_at, last_used_at, revoked_at`. Never the hash, never plaintext.
  (Masking: since we store only the hash, "masked" = a non-secret label like the
  id + created date; we cannot show last-4 of the secret unless we also store a
  short non-secret prefix at creation — acceptable to store a `prefix` column if
  the UI needs it, but that's optional and not in Spec §5, so default to id-based
  display.)
- **Revoke** (`DELETE /api/projects/:id/tokens/:tokenId`, owner-only): soft
  delete → `UPDATE ingest_tokens SET revoked_at = now() WHERE id = $tokenId AND
  project_id = $currentProjectId`. Row retained (audit-friendly, forward compat).

## 5. `PgDataAccess` + `createPostgresTraceStore` — real signature mapping

### 5.1 Two distinct objects (don't conflate them)

| Object | Implements (real, from code) | From package | Purpose |
|--------|------------------------------|--------------|---------|
| `createPgDashboardDataSource(sql, projectId)` | **`DashboardDataSource`** = `{ listTraces(): DashboardTrace[]; listModels?(): DashboardModel[]; store?: Mvp2StoreExtension }` | dashboard (type), we implement | Fed into `createReadOnlyDataAccess` to serve the 12 `/api/*` project-scoped |
| `createPostgresTraceStore(sql, projectId)` | **`ExtendedTraceStore`** = `TraceStoreReader & Mvp2StoreExtension` | sdk (type), we implement | The write+read store; its `Mvp2StoreExtension` half is what `DashboardDataSource.store` needs for /evals /cache /learning |

`createReadOnlyDataAccess(source)` (dashboard export) turns the
`DashboardDataSource` into the full `DashboardDataAccess` (getSummary,
listRequests, getRequest, listModels, compareModels, getRoutingDecision,
getEvalsOverview, getEvalRunDetail, getCacheOverview, getLearningOverview) — i.e.
all 12 endpoints — **with zero changes**. We only implement the small source.

### 5.2 `DashboardDataSource` method → SQL mapping (real field names)

`DashboardTrace` (dashboard/src/index.ts) fields we must produce:
`traceId, decisionId, chosenModel?, candidates:unknown[], reason,
attempts:{status?,latencyMs?}[], usage?, estimatedCostUsd?, estimated, latencyMs?,
status`.

```text
listTraces(): DashboardTrace[]
  SELECT trace_json FROM router_traces
  WHERE project_id = $projectId
  ORDER BY created_at DESC
  → map each trace_json (a RouterTrace) to DashboardTrace:
      { traceId, decisionId, chosenModel, candidates: t.candidates,
        reason, attempts: t.attempts.map(a=>({status:a.status,latencyMs:a.latencyMs})),
        usage: t.usage, estimatedCostUsd: t.estimatedCostUsd, estimated: t.estimated,
        latencyMs: t.latencyMs, status: t.status }
  (trace_json holds the full RouterTrace, so mapping is a field pick — no joins.)

listModels?(): DashboardModel[]
  MVP-3 has no per-project model registry table (Spec §5 has no `models` table).
  → return []  (the /models page shows empty-state; Spec §6 says Models page is
    "scoped to current project; empty state copy"). If a project has traces
    referencing models, an OPTIONAL enhancement is to derive a distinct
    chosen_model list, but that is not required and not in Spec — default [].

store?: Mvp2StoreExtension
  → the createPostgresTraceStore(sql, projectId) instance (5.3). Powers
    /evals /cache /learning via the SDK readers. If we choose not to persist
    MVP-2 eval/cache/learning per-project in MVP-3 (Spec §5 has no tables for
    them), leave `store` UNDEFINED → createReadOnlyDataAccess falls back to the
    dashboard's built-in demo/empty readers (readEvalsOverview etc.). RECOMMENDED
    for MVP-3: leave store undefined (those three pages are out of MVP-3 scope;
    Spec §6 page inventory only lists Requests/Models/Settings). Revisit in MVP-4.
```

> **Key MVP-scoping call:** Spec §5 defines tables for traces/users/projects/
> tokens only — NOT eval_runs/semantic_cache/weights_change. So the
> project-scoped control plane persists **traces** (→ Requests page) and shows
> Models empty. The /evals /cache /learning endpoints still exist (reused) but
> render the no-store demo/empty state. This keeps MVP-3 honest to its locked
> scope and avoids inventing tables the Spec didn't authorize.

### 5.3 `createPostgresTraceStore` → `ExtendedTraceStore` signature map

`ExtendedTraceStore = TraceStoreReader & Mvp2StoreExtension`. Real methods
(from `types.ts` + `storage.ts`), all project-scoped:

| Contract method (real signature) | PG implementation |
|-----------------------------------|-------------------|
| `writeTrace(trace: RouterTrace): Promise<void>` | INSERT into `router_traces` with `project_id=$projectId` (same as ingest §4.2 step 8). This is how ingest writes. |
| `listTraces(): Promise<RouterTrace[]>` | `SELECT trace_json ... WHERE project_id=$1 ORDER BY created_at DESC` → parse jsonb |
| `getSummary?(): TraceStoreSummary` | aggregate over the same scoped rows (reuse SDK's `summarizeTraces` logic or SQL COUNT/AVG). Optional. |
| `listRequests?(): StoredRequest[]` | map scoped `trace_json` → `StoredRequest` (traceId, decisionId, status, chosenModel, reason, attempts, usage, estimated, latencyMs, createdAt) |
| `getRequest?(traceId): StoredRequest?` | `... WHERE project_id=$1 AND trace_id=$2` (scope + id — cross-project id probe returns undefined, not another project's row) |
| `Mvp2StoreExtension.*` (writeEvalRun, listEvalRuns, getEvalRun, saveBaselinePointer, getBaselineRunId, writeCacheEntry, writeCacheLookup, listCacheLookups, listCacheEntries, writeWeightsChange, listWeightsChanges) | **All OPTIONAL.** Per 5.2 scoping decision, MVP-3 leaves these unimplemented → the SDK readers' `safeCall` returns empty state. Do NOT stub with fake data. |

Because every `Mvp2StoreExtension` method is optional and the SDK readers
(`dashboard-readers.ts`) guard each with `safeCall(fn, fallback)`, an
`ExtendedTraceStore` that implements only the trace-reading half is 100% valid
and the /evals /cache /learning endpoints degrade to empty gracefully. Verified
against `getEvalsOverview`/`getCacheStats`/`getLearningState` source.

### 5.4 Why the 12 endpoints need zero changes (proof)

`handleRequest` in dashboard/src/index.ts dispatches purely on `url.pathname` and
calls `data.<method>()`. It never knows where `data` came from. We construct
`data = createReadOnlyDataAccess(createPgDashboardDataSource(sql, projectId))`
per request and route the same `/api/*` paths through it. The HTML render
(`renderDashboardHtml`) is likewise a pure function of the three datasets. So the
control plane's dashboard-proxy route simply: authorize → build scoped data →
delegate to the exact same handler logic (we can either import and reuse
`createDashboard`'s handler indirectly, or re-declare the tiny path table calling
the same `DashboardDataAccess`). **Recommended:** re-expose the dashboard's route
table by importing `createReadOnlyDataAccess` + the endpoint list and writing a
thin dispatcher in `routes/dashboard-proxy.ts` that maps pathname→method,
identical to the dashboard's switch, but sourced from the scoped `dataAccess`.
(The dashboard doesn't currently export its `handleRequest`; either export it in
a tiny dashboard PR, or duplicate the ~15-line switch. Prefer exporting it to
keep one source of truth — a minimal, logic-preserving dashboard change that does
NOT add a dependency.)

### 6.1 What to assert (A13)

1. `packages/sdk/package.json` `dependencies` === `{}` (empty).
2. `packages/dashboard/package.json` deps ⊆ `{ "@adaptive-router/sdk":"workspace:*" }`.
3. `packages/cli/package.json` deps ⊆ `{ "@adaptive-router/sdk":"workspace:*" }`.
4. `packages/control-plane/package.json` deps ⊆ allowlist:
   `{ "@adaptive-router/sdk", "@adaptive-router/dashboard", "better-auth", "postgres" }`
   (workspace deps must be `workspace:*`; the two cloud deps are the only npm ones).
5. eslint `no-restricted-imports` forbids deep links into `@adaptive-router/sdk/src/**`
   (and `@adaptive-router/dashboard/src/**`).

### 6.2 Script (`scripts/check-dependency-boundary.mjs`, draft)

```js
import { readFileSync } from "node:fs"
const read = (p) => JSON.parse(readFileSync(new URL(p, import.meta.url)))
const deps = (pkg) => Object.keys(pkg.dependencies || {})
const fail = (m) => { console.error("✗ dependency boundary VIOLATED:", m); process.exit(1) }

const sdk = read("../packages/sdk/package.json")
if (deps(sdk).length) fail(`sdk must have zero deps, found: ${deps(sdk)}`)

const wsOnly = { dashboard: ["@adaptive-router/sdk"], cli: ["@adaptive-router/sdk"] }
for (const [name, allow] of Object.entries(wsOnly)) {
  const pkg = read(`../packages/${name}/package.json`)
  const extra = deps(pkg).filter((d) => !allow.includes(d))
  if (extra.length) fail(`${name} may only depend on ${allow}, found extra: ${extra}`)
  for (const d of allow) if (pkg.dependencies?.[d] && pkg.dependencies[d] !== "workspace:*")
    fail(`${name}.${d} must be "workspace:*"`)
}

const cpAllow = ["@adaptive-router/sdk", "@adaptive-router/dashboard", "better-auth", "postgres"]
const cp = read("../packages/control-plane/package.json")
const cpExtra = deps(cp).filter((d) => !cpAllow.includes(d))
if (cpExtra.length) fail(`control-plane deps not in allowlist: ${cpExtra}`)
for (const d of ["@adaptive-router/sdk", "@adaptive-router/dashboard"])
  if (cp.dependencies?.[d] && cp.dependencies[d] !== "workspace:*")
    fail(`control-plane.${d} must be "workspace:*"`)

console.log("✓ dependency boundary OK")
```

## 7. SDK-side optional reporter — zero-dependency design

**Hard rule: this adds NO npm dependency to the SDK and NO side effect when
unconfigured (A11).** It follows the MVP-2 append-only, opt-in pattern.

### 7.1 Shape (built on Node built-in `fetch`, present in Node ≥18/20/22)

New optional file `packages/sdk/src/reporter.ts` (exported from index.ts):

```ts
export type IngestReporter = {
  report(trace: RouterTrace): Promise<void>
}

export type IngestReporterOptions = {
  url: string          // e.g. https://cp.example.com/ingest/traces
  token: string        // per-project ingest token
  fetch?: typeof fetch // injectable for tests; defaults to global fetch
  onError?: (e: unknown) => void  // default: swallow (never break routing)
}

export function createIngestReporter(opts: IngestReporterOptions): IngestReporter {
  const doFetch = opts.fetch ?? fetch      // global built-in; no import, no dep
  return {
    async report(trace) {
      try {
        await doFetch(opts.url, {
          method: "POST",
          headers: { "content-type": "application/json", authorization: `Bearer ${opts.token}` },
          body: JSON.stringify(trace),
        })
      } catch (e) { opts.onError?.(e) }     // swallow by default; local routing unaffected
    },
  }
}
```

### 7.2 Wiring into `createRouter` (zero side-effect when absent)

Add an OPTIONAL `reporter?: IngestReporter` to `RouterConfig` (types.ts,
append-only — optional field, MVP-1 behavior unchanged when unset). In
`index.ts`, extend `safeWriteTrace` (or add a parallel `safeReport`) so that
after the local `store.writeTrace`, IF a reporter is configured, fire
`reporter.report(trace)` — wrapped in the same swallow-errors guard already used
for storage. When `config.reporter` is `undefined`:
- no fetch is ever called,
- no import runs (the reporter file only defines a factory; nothing executes),
- byte-for-byte identical to today (A11, and preserves `BUILTIN_WEIGHTS`
  compatibility — reporter touches only the post-decision trace, never scoring).

```text
async function safeWriteTrace(store, trace, reporter?) {
  try { await store.writeTrace(trace) } catch { /* MVP-0 rule */ }
  if (reporter) { void reporter.report(trace) }   // fire-and-forget, guarded internally
}
```

### 7.3 Why this is zero-dependency and safe

- `fetch` is a Node built-in global since Node 18 — no `node-fetch`, no npm dep.
  The SDK `package.json` `dependencies` stays `{}` → passes the §6 assertion.
- Reporter is opt-in: constructed by the SDK user and passed in. A router with no
  `reporter` does exactly what MVP-1/MVP-2 did.
- Errors are swallowed (or routed to `onError`) so a down control plane never
  breaks the caller's model call — same discipline as `safeWriteTrace`.
- Placement mirrors MVP-2: new optional file + optional config field + a guarded
  call site; nothing renamed, retyped, or made required.
- No `@types/node` needed for `fetch` (it's a lib global under
  `lib: ["ES2022","DOM"]` — the DOM lib provides `fetch`'s type). Confirmed the
  base tsconfig already includes `DOM`.

## 8. README "Design principles" wording (for team-lead to land)

> **Dependency boundary (two-tier, machine-enforced).** The core
> `@adaptive-router/sdk` carries **zero runtime dependencies** — forever. It
> embeds inside your agent runtime, so every dependency it dragged in would
> become *your* supply-chain risk and bundle weight; optional capabilities
> (embeddings, `node:sqlite`, the ingest reporter) load only through Node
> built-ins or a dynamic-import shim a bundler cannot pull in. `dashboard` and
> `cli` stay workspace-only (no npm runtime deps) too. The hosted
> `@adaptive-router/control-plane` is the **single** place standard cloud
> components live (Better-Auth, postgres.js) — because it is a service *you*
> deploy, its dependencies affect only that operator and never leak downstream.
> CI enforces this on every commit: the SDK's `dependencies` must be `{}`,
> `dashboard`/`cli` must be workspace-only, and `control-plane` may only pull
> from a fixed allowlist — otherwise the build goes red.
